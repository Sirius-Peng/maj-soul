# 雀魂复盘辅助工具（内置浏览器 + 视觉识别）Spec

## Why
雀魂官方与第三方环境缺少稳定可用的对局数据接口，导致复盘依赖手动截图与记忆。需要一个可在本地自动记录对局关键数据的工具，为后续离线分析与复盘算法提供数据基础。

## What Changes
- 提供一个桌面应用，内置浏览器内核用于打开雀魂并进行登录与对局
- 在每次对局开始时自动创建一份新的记录会话，并在对局过程中持续采集画面并输出结构化数据
- 在无可用 API 的前提下，以视觉层面（截图 + 模板匹配/轻量 OCR）识别并记录：
  - 全场已出牌（四家河）
  - 剩余牌堆（以界面剩余数指示为准；若无可靠指示则记录为 unknown）
  - 当前手牌（以自家手牌区为准）
  - 宝牌指示牌/里宝牌指示牌（可见部分）
  - 副露/暗杠/明杠等公开信息（可见部分）
  - 局/本场/供托、风场、立直棒等公共信息（可见部分）
- 识别并记录对局类型（三麻/四麻），并在会话元信息与每局元信息中落盘
- 将识别结果写入本地 SQLite 数据库，便于随时查询与后续分析功能直接消费
- 仍支持将识别结果导出为 JSON（主）与 CSV（可选），用于备份、数据迁移与离线处理
- 提供一键安装与测试验证脚本（开发者模式），自动安装所需依赖并跑通最小化测试用例

## Impact
- Affected specs: 内置浏览器宿主、画面采集、视觉识别流水线、数据模型与导出、可测试性与可重复运行
- Affected code: 新增桌面应用工程（建议 Electron），新增识别与导出模块，新增测试与工具链脚本

## ADDED Requirements
### Requirement: 桌面应用与内置浏览器
系统 SHALL 提供一个桌面应用，在应用内打开雀魂 Web 客户端并允许正常游戏对局。

#### Scenario: 打开并进入游戏
- **WHEN** 用户启动应用
- **THEN** 应用展示内置浏览器区域并可导航至雀魂登录/大厅页面
- **AND** 用户可正常登录并开始匹配/进入友人场对局

### Requirement: 对局会话自动创建
系统 SHALL 在检测到“对局开始”后自动创建新的记录会话，并在该会话结束后完成落盘导出。

#### Scenario: 自动开始记录
- **WHEN** 视觉检测判定进入新对局（从大厅/结算进入东 1 局等首局画面）
- **THEN** 系统创建新的会话目录与会话元数据（开始时间、平台、分辨率/缩放、版本信息）

#### Scenario: 自动结束记录
- **WHEN** 检测到对局结束（最终结算/返回大厅）或用户手动停止
- **THEN** 系统停止采集并导出该会话数据文件

### Requirement: 视觉采集与帧事件生成
系统 SHALL 从内置浏览器中周期性或按画面变化采集帧图像，并生成用于后续识别的“帧事件”流。

#### Scenario: 按变化触发识别
- **WHEN** 采集到的新帧与上一关键帧在指定 ROI（手牌/河/宝牌区）出现显著变化
- **THEN** 系统将该帧标记为关键帧并进入识别流程

### Requirement: 基础对局数据识别（可见信息）
系统 SHALL 在每个关键帧上识别并记录可见的对局公开信息与自家信息，输出可追溯的结构化结果。

#### Scenario: 记录自家手牌与宝牌
- **WHEN** 关键帧进入识别流程
- **THEN** 系统输出该时刻的自家手牌（0–14 张；含摸入时的第 14 张）
- **AND** 系统输出宝牌指示牌（0–N 张，按当前可见数量）

#### Scenario: 记录四家河
- **WHEN** 关键帧进入识别流程
- **THEN** 系统输出四家河的牌序列（每家 0–24 张；若牌背/遮挡则用 unknown 占位）

#### Scenario: 记录剩余牌堆
- **WHEN** 界面存在可靠的剩余牌堆数指示
- **THEN** 系统记录该数值
- **ELSE** 系统记录为 null 并附带原因（未检测到指示/置信度不足）

### Requirement: 对局类型识别（三麻/四麻）
系统 SHALL 在对局开始后识别该对局为三麻或四麻，并在会话与局级元信息中记录。

#### Scenario: 识别三麻
- **WHEN** 对局 UI 显示三人对局布局（例如玩家区/座位数为 3）
- **THEN** 系统记录对局类型为 3P

#### Scenario: 识别四麻
- **WHEN** 对局 UI 显示四人对局布局（例如玩家区/座位数为 4）
- **THEN** 系统记录对局类型为 4P

### Requirement: 数据库持久化（SQLite）
系统 SHALL 将会话、局、关键帧、识别状态快照等数据写入本地 SQLite 数据库，以便随时查询调用。

#### Scenario: 写入会话与帧数据
- **WHEN** 会话进行中产生新的关键帧识别结果
- **THEN** 系统将对应记录写入数据库（含置信度与错误字段）
- **AND** 允许在应用运行期间查询最近会话与按局/时间范围拉取数据

### Requirement: 文件落盘与导出
系统 SHALL 将会话数据落盘到本地文件系统，至少包含 JSON 导出，并支持可选 CSV 视图导出；导出内容应可由数据库记录重建。

#### Scenario: 导出会话 JSON
- **WHEN** 会话结束
- **THEN** 系统在会话目录中生成 `session.json`
- **AND** 文件内容包含：会话元信息、局列表、关键帧序列、每帧识别的状态快照、识别置信度与错误信息

### Requirement: 可测试的识别流水线
系统 SHALL 提供可重复运行的测试方式，用固定输入截图/录屏片段验证识别模块输出的稳定性。

#### Scenario: 运行最小化测试
- **WHEN** 开发者执行测试命令
- **THEN** 系统自动安装依赖并运行至少一组基准截图用例
- **AND** 输出包含：识别到的手牌/宝牌/河结果与预期比对结果

## MODIFIED Requirements
无

## REMOVED Requirements
无

## Notes / Constraints
- 识别范围以“可见信息”为准：不要求推断牌山具体构成、不要求恢复暗牌信息。
- 初期默认以固定渲染尺寸与缩放运行（例如 1280×720 或 1920×1080），通过设置页面缩放/窗口大小以获得稳定 ROI；后续再扩展自适应定位。
- 跨平台目标：macOS 与 Windows。优先选择不依赖系统级原生 OpenCV 的方案（例如 WASM 版 OpenCV + 模板匹配 + 轻量 OCR）。
- 本阶段只要求“记录数据并可查询调用（数据库）+ 可选导出”；后续“调用数据进行深度分析/可视化”的功能不在本变更范围内，但数据模型需为后续扩展预留字段。
